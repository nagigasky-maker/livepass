## 1. Why 5/4 — 矢部直氏への献呈とALL MUST DANCE™継承

### 1.1 この日付は選んだのではなく、引き継いだ

2026年5月4日は、EXPASSのプロダクトロードマップ上の都合で選ばれた日ではない。
**故・矢部直氏が紡ぎ続けた ALL MUST DANCE™ の系譜**を、CLUB ASIA TOKYOで継承する日として、この日付に EXPASS の launch を合わせた。

ALL MUST DANCE™ は、矢部直氏が日本のクラブ／ダンスミュージック文化の中核で育て上げた「踊ることの政治性」と「現場の純度」を象徴する名であり、商標であり、思想である。
EXPASS はこの名を商業的アセットとして消費する立場にはない。**継承者として、矢部直氏が現場で守ってきた価値 — 観客とアーティストの距離をゼロにする、夜を共有財産として扱う — をプロダクト設計の前提に据える。**

### 1.2 EXPASS が ALL MUST DANCE™ から引き継ぐ三つの原則

1. **現場優先（Floor First）**
   プラットフォームは、フロアの代替ではない。フロアでの体験を残し、増幅し、流通させる装置である。
   → PASSカードは、フロアに来た人にしか発行されない。オンラインでの先行販売は行わない。

2. **アーティスト主権（Artist Sovereignty）**
   売上分配・データ所有権・楽曲利用許諾は、すべてアーティスト側が握る。
   → Bandcamp式の高分配率（85%/90%）と、アーティストごとのデータエクスポート機能を Phase 1 から提供する。

3. **継承可能性（Heritability）**
   個人のキャリアではなく、シーンとカタログを次の世代に渡せる形で残す。
   → PASSカードは個人IDではなく**カタログ単位の鍵**として設計し、二次流通ではなく「保存」と「再演」を促す構造にする。

### 1.3 5/4 という日付が持つ実務的意味

| 観点 | 内容 |
|------|------|
| **法人** | 合同会社の登記完了が間に合わない場合、NOBBY個人事業主名義の口座で当日売上を一時受領（→ §7 参照） |
| **プロダクト** | PASSカード初発行日。EXPASSというブランドが世に存在を宣言する日 |
| **記録** | 当日のセットリスト・現場映像・PASS発行ログが、EXPASSのオリジナルカタログ第1弾となる |
| **コミュニティ** | 矢部直氏の系譜にいる DJ / オーガナイザー / 客が同じフロアに集まる日。これ以上のローンチ会場はない |
| **メディア** | 「故人の遺志を継いだプロダクト」という文脈は、後から作れない。5/4 を逃すと EXPASS の物語の中心を失う |

### 1.4 失敗した場合に何が残るか

5/4 当日に Stripe 決済が落ちようと、PASSカードが完売しなかろうと、**ALL MUST DANCE™ が CLUB ASIA TOKYO で行われたという事実**は残る。
EXPASS はその夜の付随物として現れ、付随物として記録される。これは退路ではなく、設計上の主従関係である。

> **EXPASS は ALL MUST DANCE™ の付録から始まる。やがて主従が逆転する日が来るかもしれないが、起源は変えない。**

## 2. Phase 1 で出す機能 / 出さない機能

### 2.1 線引きの原則

**Phase 1 (5/4〜) では、以下の三条件を満たす機能のみリリースする：**

1. **当日フロアで実証済み、または当日完結する**機能であること
2. **失敗してもオフラインで現金回収に切り替えられる**機能であること（決済系）
3. **アーティストへの分配計算が手作業でも処理できる**規模であること

これに該当しない機能は、技術的に完成していても 5/4 には出さない。

### 2.4 「出さない」を機能停止ではなく**設定**で実現する

実装済みの機能を 5/4 限定で隠すために、以下の launch flag を導入する：

```javascript
// livepass_launch_flags.js（新規作成予定）
window.LAUNCH = {
  PHASE: 1,
  ENABLE_SUBSCRIPTION: false,     // PASSサブスク
  ENABLE_TICKET_SALES: false,     // チケット販売
  ENABLE_PLAN_UPGRADE: false,     // STANDARD/PRO 課金
  ENABLE_PASS_EDITOR: false,      // PASS商品ダッシュボード
  ENABLE_FREE_COMPOSE: false,     // 自由投稿（手動キュレーションのみ）
  ENABLE_DM: false,               // DM
  ENABLE_REVENUE_DASH: false,     // 売上ダッシュボード
  ENABLE_AI_ASSIST: false,        // AI機能
  ENABLE_TIPPING: true,           // 投げ銭（Phase 1で唯一の決済）
  ENABLE_PASS_CARD_CLAIM: true,   // カード登録（Phase 1の本体）
};
```

この方式により、**コードを削除せず、当日のみ封印**できる。Phase 1b 以降は flag を順次 ON にしていく。

### 2.6 Phase 1 で「出すけど壊れててもいい」機能

## 4. Plan B — Stripe 不可時の現金 + 紙QR運用

### 4.1 Plan B 発動の判断基準（SLA定義）

Plan B への切り替えは **NOBBY のみが宣言できる**。MIZUKIが独断で切り替えない。

以下のいずれか一つが発生した時点で、NOBBY は即時 Plan B を宣言する：

| トリガー | 判断基準 |
|---------|---------|
| **Stripe 決済ページ不達** | 開場後5分以内に Stripe QRを読み込んでも決済画面が開かない状態が3回連続した場合 |
| **モバイルWi-Fi 障害** | テザリング・モバイルルーターのいずれも圏外または接続不可が5分以上続く場合 |
| **Stripe アカウント停止** | 事前の Stripe ダッシュボードで「支払い一時停止」「本人確認不備」等のアラートが出ている場合（この場合は開場前に判断） |
| **アプリ障害（登録不可）** | `/card/:id` ページが404またはFirestoreエラーで登録完了できない状態が10分以上続く場合 |

**なぜ NOBBY だけが宣言できるか**:
Stripe 決済と現金オペレーションは売上計上の方法が異なるため、後処理・税務の整合性を担保するためにオーナーが判断する。スタッフが独断で切り替えると二重計上や領収書漏れが発生する。

### 4.2 Plan B の具体手順

**事前準備物（5/4設営時に必ず持参）**:
- 紙QRシール（PASSカードIDを印刷したQRコード、台紙にシート状で印刷）: 最低300枚
- 手書き領収書帳（複写式、¥100均可）
- 現金ボックス（鍵付き）
- ペン2本（MIZUKI用・予備）
- 「現金支払い承ります」掲示用手書き札（事前作成）

**Plan B 宣言後のオペレーション**:

```
Step 1: NOBBY が「Plan B に切り替えます」と宣言
        → MIZUKIはStripeQRポスターを裏返す or 外す
        → 「現金支払い承ります」札を掲示
         ↓
Step 2: 来場者から現金を受け取る
        → PASSカードを手渡す
        → 複写領収書に「日付 / 品名:EXPASSカード / 金額 / 受領スタッフ名」を記入
        → 来場者控え（下紙）を渡す
         ↓

        → アプリ自体が落ちている場合は「48時間以内に登録できるようになります」と説明
```

### 5.3 Phase 間の移行判断基準

各 Phase への移行は「スケジュール通り」ではなく **KPI 達成を条件**とする。

| 移行 | 移行条件 |
|------|---------|
| Phase 1 → Phase 1b | PASSカード発行 150枚 以上 かつ アーティスト登録 10人以上 |
| Phase 1b → Phase 2a | MRR ¥500,000 以上 かつ STANDARD+ アーティスト 30人以上 |
| Phase 2a → Phase 2b | MRR ¥3,000,000 以上 かつ 日本国内 MAU 5,000以上 |
| Phase 2b → Phase 3 | ARR ¥60,000,000 以上 かつ 海外ユーザー比率 20%以上 |

**なぜスケジュール駆動ではなく KPI 駆動にするか**:
早期に機能を開放しすぎると、サポート負荷とセキュリティリスクが先行する。アーティストと売上のベースが整ってから次の層を追加するほうが、不具合による信頼毀損を防げる。

## 6. Bandcamp式分配の意味 — なぜ 85% / 90% なのか

### 6.1 業界標準との比較

| プラットフォーム | アーティスト分配率（実効） | 備考 |
|------------------|---------------------------|------|
| **Spotify** | **約 15〜25%**（レーベル経由） / 直契約でも 70% から税・処理費を引いた額 | ストリーム単価 ¥0.4〜¥0.7、レーベル取り分が大半 |
| **Apple Music** | 70%（プラットフォーム取り分 30%） | レーベル契約の場合さらに目減り |
| **YouTube (Content ID)** | 約 55%（広告売上） / 音楽 Premium は 45〜55% | CPM 変動大、広告ブロックで実質さらに低下 |
| **SoundCloud (Premier / Next Pro)** | 80%（プラットフォーム 20%） | 月額課金前提、無料枠は再生数連動で実効率はさらに低い |
| **Patreon** | 88〜92%（手数料 5〜12% + 決済手数料） | サブスク前提、再生数モデルではない |
| **Bandcamp** | **85%（デジタル）/ 90%（物販・閾値超）** | 売上累計でレートが上がる仕組み |
| **EXPASS** | **85%（〜累計¥50万）/ 90%（¥50万超）** | Bandcamp式の閾値モデルを継承 |

**なぜストリーミング系が低いのか**:
ストリーミングは「再生回数のロングテール」を前提とするビジネスモデルで、1回再生あたりの単価を極限まで下げないと黒字化しない。結果としてアーティストへの還元はマイクロペイメント化し、レーベル経由でない限り月¥1,000に届かないアーティストが多数派になる。

**なぜ EXPASS は 85〜90% を採用するか**:
EXPASS は再生数モデルではなく **「PASS購入」「投げ銭」「サブスク」という能動的支払いモデル**である。能動的支払いは「ファンがアーティストに直接渡す金」であり、ここから 30〜50% を抜くのは思想的に成立しない。Bandcamp が 15 年間維持してきた「能動支払い × 高分配率」の構造を、EXPASS は日本市場で再現する。

### 6.2 なぜ累計 ¥50万 を境界にしたか

**Bandcamp の閾値設計の思想**:
Bandcamp は累計売上 $5,000（約 ¥75万）を境にデジタル手数料を 15% → 10% に下げる。これは「趣味ではなく生業として継続している層」を識別する閾値であり、単なる売上ボーナスではない。

**EXPASS が ¥50万 を選ぶ理由**:

| 観点 | 内容 |
|------|------|
| **継続性の指標** | 月¥4〜5万の売上を 1 年継続できるアーティストが到達するライン。一時的バズではなく、ファンベースが定着している証拠 |
| **税務上の意味** | 日本の個人事業主の年間¥48万（基礎控除）を超える層。本格的な創作活動として税務申告が始まる規模 |
| **EXPASS 内のアーティスト構成想定** | Phase 1b 終了時で 85% / 90% 比率は **80:20** を想定。¥50万到達者は上位 2 割 |
| **会社側のキャッシュフロー** | 上位層が 90% 化することによる会社マージン低下は、下位層の量によって相殺される構造 |

**Bandcamp との違い**:
EXPASS は Bandcamp より閾値を低く設定（¥75万 → ¥50万）。理由は **日本の独立アーティスト市場が米国より小さく、年間¥50万でも十分「プロとしての継続意志」を示す指標**として機能するため。閾値を低くすることで、より早く「報われる構造」に到達できるシグナルを送る。

### 6.3 アーティスト視点の手取りシミュレーション

**前提**: 売上は PASS購入 / 投げ銭 / サブスクの合計。Stripe 決済手数料（3.6%）は EXPASS のマージン側で吸収（アーティスト分配額からは差し引かない）。

#### ケース A: 駆け出しアーティスト（累計売上 ¥30万 → ¥40万 へ伸びる）

| 月 | 月間売上 | 累計 | 適用レート | アーティスト手取り | EXPASS 取り分 |
|----|----------|------|-----------|-------------------|---------------|
| 1 | ¥30,000 | ¥30,000 | 85% | ¥25,500 | ¥4,500 |
| 2 | ¥30,000 | ¥60,000 | 85% | ¥25,500 | ¥4,500 |
| ... | ... | ... | ... | ... | ... |
| 10 | ¥30,000 | ¥300,000 | 85% | ¥25,500 | ¥4,500 |

**月 ¥10万売上の月**:
売上 ¥100,000 × 85% = **アーティスト手取り ¥85,000** / EXPASS ¥15,000

#### ケース B: 中堅アーティスト（¥50万閾値をまたぐ月）

ある月の売上が ¥1,000,000、累計が ¥45万 → ¥145万 に到達するケース：

| 区分 | 売上 | レート | 手取り |
|------|------|--------|--------|
| 閾値到達前（¥45万 → ¥50万、¥5万分） | ¥50,000 | 85% | ¥42,500 |
| 閾値到達後（¥50万 → ¥145万、¥95万分） | ¥950,000 | 90% | ¥855,000 |
| **合計** | **¥1,000,000** | — | **¥897,500** |

EXPASS 取り分: ¥102,500（実効 10.25%）

#### ケース C: 累計 ¥50万 超えた後の安定運用

累計 ¥1,000万 のアーティストが月 ¥100万 を売上：
売上 ¥1,000,000 × 90% = **アーティスト手取り ¥900,000** / EXPASS ¥100,000

**ケース C と他社比較（同じ月¥100万売上の場合）**:

| プラットフォーム | アーティスト手取り | 差額 |
|------------------|-------------------|------|
| Spotify（レーベル経由想定 20%） | ¥200,000 | -¥700,000 |
| YouTube (55%) | ¥550,000 | -¥350,000 |
| SoundCloud Premier (80%) | ¥800,000 | -¥100,000 |
| **EXPASS (90%)** | **¥900,000** | — |
| Patreon (実効 88%) | ¥880,000 | -¥20,000 |

### 6.4 競争優位性としてどう使うか

#### 6.4.1 アーティスト勧誘での使い方

**初回ピッチで提示する 1 行**:
> 「Spotify で月¥3万のアーティストが、EXPASS なら同じ熱量のファンから月¥30万を受け取れる構造を提供します」

**根拠**:
Spotify の 1 再生 ¥0.5 で月¥3万 = 月 6 万再生。同じファン層が PASS（¥3,000）を 100 人購入すると ¥300,000 × 85% = ¥255,000。**ファン数は同じでも、再生数モデルから能動支払いモデルに切り替えることで実収入が 8 倍になる。**

これは「Spotify を辞めろ」ではなく「Spotify と並行運用しろ。EXPASS は別の収益軸」という話。

#### 6.4.2 補助金申請でのフレーミング

| 補助金 | EXPASS の高分配率を訴求する文脈 |
|--------|-------------------------------|
| **持続化補助金（小規模事業者）** | 「日本の独立アーティストの所得改善に直接寄与する販路」として、創作者の事業継続性に対する社会的意義を強調 |
| **IT導入補助金** | 「アーティスト個人の収益管理 IT インフラ」として、従来レーベルが独占していた決済・分配を個人事業主に開放する DX 事例 |
| **文化庁補助金（文化芸術活動継続支援）** | 「文化の担い手が文化で食えるための基盤」として、創作者経済（クリエイターエコノミー）の日本版モデル |
| **東京都助成（産業労働局・スタートアップ支援）** | 「東京発のクリエイター主権プラットフォーム」として、地域発信力強化 |

#### 6.4.3 投資家ピッチでの位置付け

**投資家からの典型的な懸念**:
> 「分配率 90% で会社は儲かるのか？」

**回答スクリプト**:
1. **第 1 軸の薄マージンは「信頼の固定コスト」と捉える**
   分配率を業界最高水準に固定することで、Spotify / Apple とは別カテゴリとして認知される。アーティストが「乗り換える理由」ではなく「並行する理由」になり、ロックインを生む。
2. **第 2 軸の収益でカバーする**
   - サブスクリプション（アーティスト課金、Phase 1b 以降）: STANDARD ¥1,500〜2,000/月、PRO ¥4,000〜5,000/月
   - PASS二次流通ロイヤリティ（Phase 2 以降）: 5〜10% を会社取り分
   - 企業スポンサー / BRAND プラン（Phase 2b 以降）: 月数十万円単位
   - イベント運営（ORGANIZER プラン、Phase 2a 以降）: 売上 GMV ベース
3. **GMV モデルとして見れば成立する**
   分配率 10% でも、GMV ¥10億 なら会社売上 ¥1億。Bandcamp は 2022 年に GMV $1B（約¥150億）を達成し、Epic Games に買収された。EXPASS は日本市場で同じ構造を狙う。

### 6.5 リスクと緩和策

| リスク | 内容 | 緩和策 |
|--------|------|--------|
| **会社マージンが薄い** | 分配率 85〜90% を取り分にすると、Stripe 手数料（3.6%）後の実効マージンは **6.4〜11.4%**。スケールするまでは黒字化が遠い | 第 2 軸（サブスク・スポンサー・二次流通ロイヤリティ）を Phase 1b から段階投入。Phase 2 以降の主力収益に育てる |
| **同業他社が分配率で並ぶ** | Bandcamp が日本展開を強化、または国内競合が同率を打ち出す可能性 | EXPASS の差別化は「分配率 + 矢部直氏由来のシーン正統性 + 日本独自のクラブ／カルチャー文脈」。価格だけでは競合不可能なレイヤーで戦う |
| **アーティストが閾値超えを実感する前に離脱** | 累計 ¥50万 到達まで 1〜2 年かかるアーティストが多く、その間に他プラットフォームに移る | Phase 1b でアーティスト向けダッシュボードに「累計売上 / 閾値までの残額」可視化を実装。心理的継続インセンティブを作る |
| **会社維持コストが想定を超える** | サポート対応・サーバー・税務処理コストが GMV 比例で増える | Phase 2 以降にサポート FAQ / セルフサービス化を本格投資。手数料の中の固定費比率を下げる |
| **Stripe 手数料の値上げ** | Stripe が日本市場で料率改定する可能性 | Phase 2 で代替決済（Square、GMO）の併用を検討。複数決済の冗長化 |

### 6.6 一行で説明する EXPASS の分配構造

> **「ファンが¥1,000払えば、アーティストに¥850 から¥900 届く。中抜きしない。これが原則。」**

この一文は、補助金申請書・投資家資料・アーティスト勧誘・メディア取材のどれでも使える。EXPASS の根幹であり、5/4 以降に揺らがせない。
